# 安全边界

## 安全目标

这个项目的目标是避免将私密文章正文以普通明文内容的形式保存在服务端发布层。

它不承诺 DRM，也不承诺在浏览器解密后仍能完全控制读者行为。

## 预期安全边界

私密文章数据应拆分为：

- 公开元数据
- 加密后的私密正文 bundle

服务端保存：

- 密文
- 算法元数据
- 公开文章元数据
- 站点恢复私钥（Secret：`halo-private-posts-site-recovery`）

浏览器临时持有：

- 解锁时输入的访问密码
- 解密后的正文内存数据

浏览器不应持久化：

- localStorage 中的解密正文
- localStorage 中的访问密码
- 插件逻辑产生的明文正文缓存

## 这个插件应该防护什么

- 普通文章存储流程中的明文泄露
- 服务端直接检查私密文章正文
- 路由切换或长时间空闲后的残留解锁状态

## 这个插件不防护什么

- 截图
- 成功解锁后的复制粘贴
- 拥有页面访问权限的浏览器扩展
- 已被攻陷的用户设备
- 弱访问密码

## 设计规则

- 在浏览器端解密私密正文
- 只在内存中保存解密结果
- 在页面离开、标签页隐藏和空闲超时后清理解锁状态
- 明确区分公开元数据和私密正文
- 不在 Halo 服务端保存访问密码
- 平台恢复私钥只能保留在服务端，不能下发到浏览器

## 当前恢复边界

当前版本已经明确采用平台托管恢复：

- 服务端保存站点恢复私钥
- 拥有后台恢复权限的一方可以重置访问口令
- 这意味着服务端、数据库备份和 Secret 备份都属于安全边界的一部分

因此当前版本不应再宣称：

- 零托管恢复
- 平台绝对无法恢复正文
- 作者自己之外没人能恢复

这是产品取舍，不是实现偏差。

## 恢复方向

如果后续继续演进，应在当前平台恢复模型上补更细的权限控制、审计和运维说明，而不是模糊当前边界。
