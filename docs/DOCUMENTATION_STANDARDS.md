# 文档规范

本文档定义仓库内文档的职责边界和更新规则。目标是让实现、行为契约、运维步骤和测试门槛分开表达，避免同一信息散落在多个文件里互相漂移。

## 基本原则

1. 文档必须区分 `实现细节`、`行为契约`、`运维步骤`、`测试门槛`。
2. 任何对外可感知的行为变化，必须在同一提交里更新对应文档。
3. 文档描述必须使用精确名词，不使用“可能”“大概”“类似”这类模糊措辞。
4. 对路由、任务名、文件路径、资源名的描述，必须使用代码里的真实值。
5. 如果一个行为已经被定义为契约，其他文档只能引用它，不能再写第二份语义略有差异的版本。

## 文档职责

### `README.md`

- 面向首次进入仓库的开发者或使用者。
- 只保留高层介绍、快速入口和文档导航。
- 不承载深入的行为契约和长篇运维细节。

### `docs/ARCHITECTURE.md`

- 说明系统分层、资源关系、主流程和边界。
- 回答“系统为什么这么组织”。
- 不写发布流程和人工回归步骤。

### `docs/MAINTENANCE.md`

- 说明当前实现约束、主入口、常见误区和维护注意事项。
- 回答“改代码时哪些约束不能碰”。
- 不作为唯一的行为标准来源。

### `docs/SAVE_ENCRYPTION_CONTRACT.md`

- 是文章保存、加密、密文同步、失败语义的唯一行为契约。
- 回答“什么行为算正确”。
- 涉及保存入口、加密同步和 UI 可见性的改动，必须同步更新这里。

### `docs/SMOKE_TEST.md`

- 定义发布前 smoke、人工回归和保存回归矩阵。
- 回答“上线前最少要验什么”。
- 不重复解释内部实现原理。

### `docs/QUALITY_GATES.md`

- 定义改动类型与必须执行的测试/构建入口。
- 回答“这次改动至少要跑哪些检查”。

### `docs/OPERATIONS.md`

- 面向站点管理员，定义安装、升级、回滚、卸载。
- 回答“怎么上线和出问题后怎么处理”。

## 更新规则

### 改动保存或加密行为时

必须同时检查并按需更新：

- `docs/SAVE_ENCRYPTION_CONTRACT.md`
- `docs/SMOKE_TEST.md`
- `docs/MAINTENANCE.md`

### 改动构建、测试或 CI 入口时

必须同时检查并按需更新：

- `docs/QUALITY_GATES.md`
- `README.md`
- `.github/workflows/*.yml` 中对应入口说明

### 改动部署或恢复流程时

必须同时检查并按需更新：

- `docs/OPERATIONS.md`
- `docs/RECOVERY_MODES.md`
- `docs/SMOKE_TEST.md`

## 写作要求

1. 用“入口 + 预期结果”的形式描述检查项。
2. 用“请求方法 + 路径 + 请求体类型”的形式描述保存链路。
3. 用“谁是真源、谁是镜像”的形式描述数据关系。
4. 出现版本、路径、命令时，必须写完整值。
5. 若文档描述的是限制条件，必须明确“不允许什么”。

## 合并前最低要求

1. 变更涉及的文档已同步更新。
2. `README.md` 文档导航能指向新的规范文档。
3. 自动化检查入口与文档中的命令名称一致。
